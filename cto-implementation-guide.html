01

סקירה כללית

הבונקר הוא אולפן פודקאסט המתמחה בהפקה, עריכה ותפעול של תכני וידאו ארוכים-קצרים: פרק פודקאסט מלא + רילס נגזרים. המערכת שבמסמך זה היא פלטפורמת ניהול פנימית שלוקחת את ההפקה מהקלטה עד מסירה ללקוח — בלי גוגל-דוקים פרושים על המסך ובלי לוחות פיזיים על הקיר.

הבעיה שאנחנו פותרים

היום, ניהול 16-40 פרקים בחודש מבוסס על וואטסאפ + Drive + תפילות. כשלקוח שולח תיקונים — הם מתפזרים. כשהוא רוצה לסמן 5 רילס מתוך פרק — הוא כותב במייל "דקה 12:40 עד 13:25". המידע לא מובנה, גרסאות מתבלבלות, ולפעמים פרק מתמסר לפני שהלקוח אישר אותו.

הפתרון

פלטפורמה אחת שמרכזת את כל מחזור החיים של הפרק:

  • פורטל לקוח — נגן וידאו מתקדם עם סימון Timeline ל-Reels והערות ממוקדות-זמן
  • דאשבורד צוות — Kanban של כל הפרקים והרילס בכל הסטטוסים
  • אוטומציה — Make.com שולח התראות, מסנכרן עם Monday, מטריג WhatsApp
  • ספרייה — Bunny Stream מארח את הוידאו, סטרימינג מהיר עם signed URLs
ליבת המוצר הלב של המערכת הוא פורטל הלקוח. ההצלחה נמדדת בכך שלקוח יקבל לינק, יסמן 5 רילס + יכתוב הערות, ויחזיר תשובה בלי לפתוח אף ערוץ אחר.

02

כללי הזהב — CTO Rules

הכללים הבאים אינם מוצעים — הם חובה. כל החלטה ארכיטקטונית נמדדת מולם. סטייה דורשת אישור מאלון בלבד.

⚡ NON-NEGOTIABLE RULES
  1. Bunny Stream — ולא חברה אחרת. כל וידאו בפלטפורמה מסוטרם דרך Bunny. לעולם לא מ-Drive embed, לעולם לא ב-S3 ישיר. Signed URLs בלבד. Egress של Bunny זול וצפוי — לא מקבלים הפתעות.
  2. Supabase — מקור האמת היחיד. מצב המערכת חי ב-Postgres של Supabase. Monday משקף את הסטטוסים אבל לא מנהל אותם. אם Supabase ו-Monday לא תואמים, Supabase ניצח.
  3. Make.com — לא Code. כל אוטומציה (התראות, סנכרון, תזכורות) רצה ב-Make. לא כותבים cron בקוד, לא setTimeout, לא queues. אם Make לא מספיק — נשדרג ל-Trigger.dev, אבל לא לפני שזה הופך לבעיה אמיתית.
  4. RTL ועברית — מהיום הראשון. לא מפתחים LTR ואז "נתאים אחר כך". כל קומפוננטה נבנית RTL-native. Heebo כפונט ברירת מחדל. בדיקה ב-Safari (לקוחות = מובייל = iOS).
  5. Token Access ללקוח — לא חשבונות. לקוח פותח לינק חד-פעמי ועובד. אין סיסמה ב-MVP. הטוקן יש לו תאריך פג תוקף (30 יום default) ו-revoke יכולת ב-Admin.
  6. טיפוסי TypeScript בכל מקום. strict mode מופעל. אין any ללא הצדקה כתובה. Zod לוולידציה של API.
  7. אין mock data בפרודקשן. Seed scripts לסביבת dev בלבד. ב-prod — נתונים אמיתיים או null.
חשוב הכלל החשוב מכולם — אם אתה חוסך זמן עכשיו בקיצורי דרך, אתה משלם פי 3 בעוד חודשיים. בנו את התשתית נכון מההתחלה.

זרימת מידע

  1. העלאת פרק (Admin) Admin מעלה קובץ דרך Bunny TUS upload → מקבל video_id → רושם בטבלת episodes ב-Supabase
  2. שינוי סטטוס Admin משנה ל-"sent_v1" → Supabase trigger מפעיל webhook ל-Make
  3. Make מפיץ יצירת share token → שליחת מייל + WhatsApp ללקוח → עדכון Monday item
  4. לקוח פותח לוחץ לינק → Next.js validates token → טוען Bunny signed URL → מציג נגן + Timeline
  5. לקוח שולח פידבק Reels markers + comments → POST ל-API → Supabase insert → trigger Make → התראה לאדמין
  6. ייצוא לעורך Admin לוחץ "ייצא Reels" → מקבל JSON/CSV עם timestamps + שמות לעבודה ב-Premiere

מודל נתונים

סכמת Postgres ב-Supabase. כל הטבלאות עם RLS מופעל. השדות עם NOT NULL מסומנים. תאריכים ב-UTC, מומרים ב-UI לישראל.

טבלת customers (לקוחות)

CREATE TABLE customers (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name            text NOT NULL,
  email           text NOT NULL UNIQUE,
  whatsapp        text NOT NULL,  -- 9725XXXXXXXX format
  package         text NOT NULL CHECK (package IN ('photo_only', 'photo_edit', 'retainer')),
  retainer_active boolean NOT NULL DEFAULT false,
  notes           text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);

טבלת episodes (פרקים)

CREATE TYPE episode_status AS ENUM (
  'recorded', 'reviewing',
  'sent_v1', 'revisions_v1',
  'sent_v2', 'revisions_v2',
  'sent_v3', 'approved', 'archived'
);

CREATE TABLE episodes (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id         uuid NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
  title               text NOT NULL,
  episode_number      int,
  recorded_at         date NOT NULL,
  status              episode_status NOT NULL DEFAULT 'recorded',
  current_version     int NOT NULL DEFAULT 1 CHECK (current_version <= 3),
  bunny_video_id      text,            -- current version's video
  share_token         text UNIQUE,    -- nanoid(32)
  share_expires_at    timestamptz,
  created_at          timestamptz NOT NULL DEFAULT now(),
  updated_at          timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_episodes_customer ON episodes(customer_id, recorded_at DESC);
CREATE INDEX idx_episodes_status ON episodes(status);
CREATE INDEX idx_episodes_share_token ON episodes(share_token) WHERE share_token IS NOT NULL;

טבלת episode_versions (גרסאות וידאו)

CREATE TABLE episode_versions (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  episode_id      uuid NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
  version_number  int NOT NULL CHECK (version_number BETWEEN 1 AND 3),
  bunny_video_id  text NOT NULL,
  duration_seconds int,
  sent_at         timestamptz,
  approved_at     timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now(),
  UNIQUE(episode_id, version_number)
);

טבלת reels (קטעי רילס)

CREATE TYPE reel_status AS ENUM (
  'selected', 'editing',
  'sent_v1', 'revisions_v1',
  'sent_v2', 'revisions_v2',
  'sent_v3', 'approved'
);

CREATE TABLE reels (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  episode_id          uuid NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
  order_num           int NOT NULL,  -- auto-computed by start_time
  title               text NOT NULL,
  start_time_seconds  numeric(10,2) NOT NULL,
  end_time_seconds    numeric(10,2) NOT NULL CHECK (end_time_seconds > start_time_seconds),
  status              reel_status NOT NULL DEFAULT 'selected',
  bunny_video_id      text,            -- once edited
  created_by          text NOT NULL CHECK (created_by IN ('client', 'admin')),
  created_at          timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_reels_episode ON reels(episode_id, order_num);

טבלת feedback_items (הערות)

CREATE TABLE feedback_items (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  episode_id          uuid NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
  version_id          uuid NOT NULL REFERENCES episode_versions(id),
  type                text NOT NULL CHECK (type IN ('timestamped', 'general')),
  timestamp_seconds   numeric(10,2),   -- NULL for general
  comment             text NOT NULL,
  resolved            boolean NOT NULL DEFAULT false,
  created_at          timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_feedback_episode_unresolved
  ON feedback_items(episode_id) WHERE resolved = false;

RLS Policies (Row Level Security)

CRITICAL Admin authenticated משתמש → גישה מלאה. לקוח דרך share_token → גישה רק לפרק הספציפי. אף פעם לא לסמוך על הקליינט לסנן את הגישה. 
-- Admin sees everything
CREATE POLICY "admin_full_access" ON episodes
  FOR ALL TO authenticated
  USING (auth.jwt() ->> 'role' = 'admin');

זרימת לקוח (Client Journey)

החוויה החשובה ביותר במערכת. הלקוח לא מתכוון להישאר חצי שעה — הוא רוצה לסיים בתוך 10-15 דקות.

  1. קבלת התראה (Email + WhatsApp) "היי [שם], הפרק "[שם פרק]" שלך מוכן לצפייה. לחץ כאן: bunker.app/p/abc123" — שני ערוצים במקביל, השני נשלח 2 דקות אחרי הראשון
  2. פתיחת הפורטל הלינק טוען את הפורטל עם הוידאו כבר מוכן לנגינה. ברכה אישית: "שלום [שם]" + הוראות קצרות "סמן את הקטעים שמדברים אליך + השאר הערות תוך כדי"
  3. צפייה והערכה הלקוח מתחיל לצפות. הוא יכול להאיץ ל-1.5x או 2x להערכה ראשונית. רואה ציר זמן מלא. כשהוא רוצה — לוחץ I ל-Mark-In, O ל-Mark-Out — סימן ריל נשמר
  4. סימון Reels אחרי סימון start/end — נפתח קונפרם קטן: שדה "שם הריל" (אופציונלי, ברירת מחדל "ריל #2"). יכול לסמן עוד ועוד. הרילס מתמספרים אוטומטית לפי סדר ה-Timeline (גם אם הוסיף בלי סדר)
  5. השארת הערות לחיצה כפולה על Timeline → פותחת הערה ממוקדת-זמן ("ב-12:34 — תוריד את הקטע על השכן"). או — שדה הערות כלליות בתחתית ("הפתיח קצת ארוך")
  6. שליחת הפידבק כפתור גדול אדום "סיימתי - שלח לבונקר". אם יש > 0 reels או > 0 הערות — נשלח. אחרת — מציע "האם אתה רוצה לאשר את הפרק כמו שהוא?"
  7. קבלת אישור מסך אישור עם סיכום: "שלחת 5 רילס + 3 הערות. נעדכן אותך כשהגרסה הבאה מוכנה." + ועדכון WhatsApp לאלון בו-זמנית

זרימת אדמין (Admin Journey)

שגרת בוקר

  1. פתיחת הדאשבורד רואה Kanban: "ממתין לבקרה" / "אצל לקוח" / "תיקונים התקבלו" / "מוכן למסירה" + מספרים
  2. בדיקת התראות "שני לקוחות החזירו פידבק", "פרק [X] נמצא 4 ימים אצל הלקוח — שולח תזכורת אוטומטית מחר"
  3. ייצוא תור עורך כפתור "ייצא Reels לעריכה" → מקבל JSON עם start/end + שמות לכל הרילס שב-status='selected'

העלאת פרק חדש

  1. יצירת רשומת פרק "+ פרק חדש" → בוחר לקוח → ממלא שם פרק, מספר, תאריך הקלטה
  2. העלאת קובץ Drag & drop של הקובץ → Bunny TUS upload (תומך resume) → progress bar → קבלת video_id
  3. סקירה פנימית אלון עובר על הוידאו, סטטוס "reviewing"
  4. שליחה ללקוח כפתור "שלח ללקוח" → יצירת share_token → Make scenario מופעל → Email + WhatsApp נשלחים → סטטוס "sent_v1"
  5. מעקב פידבק כשהלקוח מסיים — התראה ב-WhatsApp לאלון. הוא רואה את כל ה-Reels וההערות במקום אחד
  6. קבלת תיקונים / אישור אם יש פידבק — סטטוס "revisions_v1", מסמן לעורך מה לעדכן. אם אושר — "approved", הפרק נסגר

דוגמת קוד — POST /api/p/[token]/reels

// app/api/p/[token]/reels/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { z } from 'zod';
import { createServerClient } from '@/lib/supabase';

const ReelInputSchema = z.object({
  title: z.string().min(1).max(100),
  start_time_seconds: z.number().min(0),
  end_time_seconds: z.number().min(0),
});

const RequestSchema = z.object({
  reels: z.array(ReelInputSchema).min(1).max(20),
});

export async function POST(
  req: NextRequest,
  { params }: { params: { token: string } }
) {
  const body = await req.json();
  const parsed = RequestSchema.safeParse(body);

  if (!parsed.success) {
    return NextResponse.json({ error: 'Invalid input' }, { status: 400 });
  }

  const supabase = createServerClient();
  const { data: episode } = await supabase
    .from('episodes')
    .select('id, share_expires_at')
    .eq('share_token', params.token)
    .single();

  if (!episode || new Date(episode.share_expires_at) < new Date()) {
    return NextResponse.json({ error: 'Token expired' }, { status: 403 });
  }

  // Sort by start_time and assign order_num
  const sorted = [...parsed.data.reels].sort((a, b) => a.start_time_seconds - b.start_time_seconds);
  const withOrder = sorted.map((r, i) => ({ ...r, order_num: i + 1, episode_id: episode.id, created_by: 'client' }));

  const { error } = await supabase.from('reels').insert(withOrder);
  if (error) return NextResponse.json({ error }, { status: 500 });

  return NextResponse.json({ data: { created: withOrder.length } });
}

// lib/bunny/signed-url.ts
import { createHash } from 'crypto';

export function getSignedVideoUrl(
  videoId: string,
  expiresInSeconds: number = 3600
): string {
  const libraryId = process.env.BUNNY_LIBRARY_ID!;
  const securityKey = process.env.BUNNY_SECURITY_KEY!;
  const expiry = Math.floor(Date.now() / 1000) + expiresInSeconds;
  const hash = createHash('sha256')
    .update(securityKey + videoId + expiry)
    .digest('hex');
  return `https://iframe.mediadelivery.net/embed/${libraryId}/${videoId}?token=${hash}&expires=${expiry}`;
}
